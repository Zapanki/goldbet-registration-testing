# QA Report: Goldbet Registration / Authorization

| Затраченное время | 2 часа |

## Ограничения проверки

- Тестирование проводилось методом чёрного ящика на UI-уровне публичной формы регистрации / авторизации.

## Примечание по окружению

При открытии `https://goldbet.io/registration` страница редиректит на зеркало вида:

`https://goldbet*.top/registration?currency=EUR&method=email&mode=sign-up&bonus=171`

Зеркало менялось между запусками (`goldbet7ko.top`, `goldbet8az.top`, `goldbet10io.top`, `goldbet17fs.top`, `goldbet20az.top`, `goldbet22fh.top`). Проверки выполнялись на текущем открывшемся зеркале.

Базовый URL `https://goldbet.io/registration` в текущем окружении открыл греческую локаль (`EL`), несмотря на `en-US` в браузере. Английский UI доступен явно через `/en/registration`.

## Исследовательское тестирование

Проверенные основные пользовательские сценарии:

- Открытие страницы регистрации по URL из задания.
- Редирект на актуальное зеркало `goldbet*.top`.
- Регистрация через E-mail до pre-flight CAPTCHA.
- Регистрация через Phone до UI-этапа ввода телефона.
- Проверка верхнего блока социального входа `Facebook / Google / Telegram`.
- Визуальное переключение вкладок `E-mail / Phone / Один клик`, включая состояние формы после CAPTCHA и legal validation.
- Переход из Sign Up в Sign In и обратно.
- Проверка блока валюты: `Euro`, `US Dollar`, обработка невалидной валюты в URL.
- Проверка блока welcome bonus и обработки невалидного `bonus` в URL.
- Проверка кнопки добавления promo code.
- Проверка 18+ / Terms consent на UI-уровне.
- Проверка submit с невалидным E-mail (ввод спецсимволов и HTML-тегов) до CAPTCHA.
- Проверка submit при снятом 18+ / Terms consent.
- Проверка повторного клика `Sign Up` на замедленной сети.
- Проверка ссылок `Terms and Conditions`, `Privacy, Management of Personal Data`, `Cookie Policy`.

## Функциональное тестирование

### Что работает корректно

- Страница открывается и отображает форму регистрации.
- Форма содержит верхний блок социального входа (`Facebook`, `Google`, `Telegram`) и три вкладки основных методов регистрации: `E-mail`, `Телефон`, `Один клик`.
- При переходе на `/en/registration` UI отображается на английском языке: `E-mail`, `Phone`, `One click`, `Sign Up`, `Sign In`.
- Пустая форма E-mail-регистрации недоступна для обычного пользователя: кнопка `Sign Up` находится в disabled-состоянии.
- При принудительном снятии `disabled` через DOM и клике по пустой форме на UI-уровне нет перехода дальше: CAPTCHA не появляется, URL не меняется.
- После ввода значений в поля E-mail и Password кнопка `Sign Up` становится активной.
- Вкладки `E-mail`, `Phone`, `One click` визуально переключаются без перезагрузки страницы.
- При переключении на Phone отображается поле телефона и код страны `+30`.
- Переход `Sign In` меняет URL на `mode=sign-in` и открывает форму входа.
- В форме входа доступны E-mail/Phone, ссылка восстановления пароля и социальный вход.
- Ссылки Terms/Privacy/Cookie присутствуют и ведут на релевантные страницы текущего зеркала. Privacy page содержит раздел Cookies.
- Невалидные URL-параметры нормализуются: `currency=ABC`, `bonus=999999`, `method=admin` возвращаются к валидным `EUR`, `171`, `email`. По UI-наблюдению строка в браузере меняется после загрузки страницы; HTTP 301/302 отдельно не проверялся.
- Валидная подмена `currency=USD` отображает `US Dollar` и сохраняет hidden/input значение `USD`.
- При double click по `Sign Up` на замедленной сети кнопка на UI-слое заменяется на CAPTCHA-блок, повторно нажать `Sign Up` визуально нельзя.

### Что работает некорректно

- Кнопки социальных методов авторизации в верхнем блоке `Продолжить через` отображаются как icon-only buttons без видимого текста, `aria-label` и `title`.
- При потере сети во время CAPTCHA система показывает некорректную ошибку `Неправильная капча. Пожалуйста, попробуйте еще раз` вместо сообщения об отсутствии сети.
- Форма регистрации раскрывает существование зарегистрированного E-mail: при вводе существующего E-mail и неверного пароля отображается ошибка `Электронная почта уже существует`.
- 18+ / Terms consent реализован кастомным `div data-active="true/false"` без стандартного `input[type=checkbox]`, `role="checkbox"` и явного доступного состояния. Для обязательного legal control это делает состояние менее прозрачным на UI-уровне.
- Редирект на зеркало происходит без пользовательского статуса или объяснения. На медленном соединении это может выглядеть как нестабильная загрузка страницы, хотя фактически выполняется маршрутизация на актуальный домен.
- При попытке регистрации без обязательного 18+ / Terms consent система блокирует отправку на клиентской стороне, но не показывает явную ошибку валидации. После CAPTCHA пользователь остается с блоком `Проверено` без объяснения причины отказа.
- На мобильной ширине менее 400px форма не помещается во viewport: правый край контейнера и текст 18+ consent обрезаются.

### Что вызывает вопросы

- Какой ожидаемый UX для повторного запроса SMS / повторного прохождения CAPTCHA: должен ли UI показывать cooldown-таймер, disabled-состояние или нейтральный текст статуса?
- Должен ли выбранный welcome bonus / currency иметь отдельный визуальный highlight после редиректа на зеркало или изменения URL-параметров?

## Баг-репорты

Ниже шесть подтвержденных дефектов.

### BUG-01: [Registration / Social Login] Icon-only social login buttons have no accessible names

| Поле | Значение |
| --- | --- |
| Severity | Medium |
| Priority | Low |
| Область | Registration / Social login / Accessibility |
| Статус | Reproducible |

**Шаги воспроизведения:**

1. Открыть `https://goldbet.io/en/registration`.
2. Найти верхний блок социального входа `Continue with` / `Продолжить через`.
3. Проверить icon-only кнопки `Facebook`, `Google`, `Telegram`.

**Фактический результат:**

Social login buttons в верхнем блоке `Продолжить через` отображаются как icon-only кнопки без видимого текста, `aria-label` и `title`. Пользователь видит только иконки провайдеров, а assistive technologies не получают понятное имя действия.

**Ожидаемый результат:**

Каждая кнопка социального входа должна иметь доступное имя, например `Continue with Google`, `Continue with Facebook`, `Continue with Telegram`. Это можно реализовать через видимый текст, `aria-label` или другой корректный accessible name.

**Влияние:**

Пользователи screen reader и других assistive technologies не могут однозначно понять назначение icon-only кнопок. Для целевой аудитории онлайн-казино это низкий sprint-priority дефект, но его стоит зафиксировать как accessibility debt.

### BUG-02: [Routing] Mirror redirect has no visible loading or status state

| Поле | Значение |
| --- | --- |
| Severity | Low |
| Priority | Medium |
| Область | Registration / Mirror routing / UX |
| Статус | Reproducible |

**Шаги воспроизведения:**

1. Открыть URL из задания: `https://goldbet.io/registration`.
2. Дождаться редиректа на актуальное зеркало `goldbet*.top`.
3. Повторить открытие несколько раз или на замедленной сети.

**Фактический результат:**

Страница переводит пользователя на зеркало без явного пользовательского статуса, объяснения или стабильного loading-состояния. Между запусками зеркало меняется.

**Ожидаемый результат:**

Во время маршрутизации UI должен показывать нейтральный брендированный loading-state без акцента на технической маршрутизации: например, анимацию логотипа, `Securely connecting...` или `Loading gaming lobby...`.

**Влияние:**

Низкий функциональный риск, но заметное UX-трение на первом шаге воронки. Пользователь может воспринять редирект между доменами как ошибку загрузки; при этом UI-текст не должен дополнительно акцентировать техническую механику маршрутизации.

### BUG-03: [Security] User Enumeration via registration form

| Поле | Значение |
| --- | --- |
| Severity | Medium |
| Priority | Medium |
| Область | Registration / Auth Flow / Security |
| Статус | Reproducible |

**Шаги воспроизведения:**

1. Открыть `https://goldbet.io/registration` или актуальное зеркало.
2. Заполнить поле E-mail адресом электронной почты, который уже зарегистрирован в системе.
3. Ввести корректный пароль от этого существующего аккаунта.
4. Пройти CAPTCHA-проверку и отправить форму.
5. Повторить сценарий с тем же зарегистрированным E-mail, но с неверным паролем.

**Фактический результат:**

Система по-разному реагирует на ввод данных в форме регистрации:

- Если пароль верный, пользователя авторизует и перенаправляет в личный кабинет / dashboard.
- Если пароль неверный, система явно сообщает: `Электронная почта уже существует`.

Такое поведение раскрывает факт существования E-mail в базе без успешной авторизации.

**Ожидаемый результат:**

Форма регистрации не должна раскрывать, зарегистрирован ли конкретный E-mail. При попытке использовать существующий E-mail с неверным паролем система должна показывать нейтральную ошибку или переводить пользователя в корректный login/recovery flow без явного подтверждения существования аккаунта.

**Влияние:**

Злоумышленник может автоматизированно просеивать списки E-mail адресов и определять, кто из пользователей зарегистрирован в казино. Это повышает риск targeted phishing, account takeover attempts и последующего credential stuffing уже через штатные auth-механизмы.

### BUG-04: [UX/Error Feedback] Silent feedback after attempting registration without 18+ consent

| Поле | Значение |
| --- | --- |
| Severity | Medium |
| Priority | Low |
| Область | Registration / Legal Consent / UX Feedback |
| Статус | Reproducible |

**Шаги воспроизведения:**

1. Открыть `https://goldbet.io/registration` или актуальное зеркало.
2. Ввести валидные, ранее не зарегистрированные E-mail и пароль.
3. Снять обязательный чекбокс `18+ / Terms consent`.
4. Нажать кнопку `Sign Up`.
5. Успешно пройти появившуюся CAPTCHA-проверку.

**Фактический результат:**

Silent feedback на UI-уровне. После CAPTCHA пользователь остается на форме без понятного сообщения, почему регистрация не продолжается. Кнопка `Sign Up` исчезает, заменяясь на некликабельный блок CAPTCHA `Проверено`; пользователь не получает обратной связи о том, что сценарий остановлен из-за снятого 18+ consent.

**Техническая квалификация:**

Проверка через Network tab показывает, что после успешного CAPTCHA `redeem` POST-запрос на регистрацию `graphql?op=UserRegistration...` не отправляется. Для сравнения, при успешной регистрации с включенным 18+ consent такой POST-запрос присутствует.

Блокировка происходит на клиентской стороне: фронтенд-скрипт видит снятый чекбокс, останавливает registration flow, но не выводит UI-уведомление об ошибке валидации.

**Ожидаемый результат:**

Снимаемый 18+ checkbox лучше заменить на статичный legal consent text под кнопкой `Sign Up`: `Нажимая Sign Up, вы подтверждаете, что вам есть 18 лет и вы принимаете Terms and Conditions`. Тогда пользователь не сможет случайно создать invalid state, а юридическое подтверждение останется частью действия submit.

**Влияние:**

Это edge case: чекбокс предзаполнен, поэтому большинство реальных пользователей не будут снимать его целенаправленно. Тем не менее пользователь, случайно снявший чекбокс или не заметивший его состояния, остается без понятной причины отказа.

**Выбранное UX-решение:**

Не чинить ошибочное состояние чекбокса через дополнительные validation states, а убрать саму возможность снять предзаполненное legal-согласие. Это полностью убирает класс ошибок, связанных с silent validation failure по removable legal control.

### BUG-05: [Error Handling] Offline CAPTCHA validation is shown as incorrect CAPTCHA

| Поле | Значение |
| --- | --- |
| Severity | Minor |
| Priority | Medium |
| Область | Registration / CAPTCHA / Error Handling |
| Статус | Reproducible |

**Шаги воспроизведения:**

1. Открыть `https://goldbet.io/registration` или актуальное зеркало.
2. Заполнить регистрационную форму валидными данными.
3. Перейти к CAPTCHA-проверке.
4. В DevTools включить режим `Offline` перед отправкой / валидацией CAPTCHA.
5. Попробовать завершить CAPTCHA / submit.

**Фактический результат:**

При отсутствии сети отображается ошибка `Неправильная капча. Пожалуйста, попробуйте еще раз`. Пользователь получает сообщение, будто CAPTCHA решена неверно, хотя реальная причина — отсутствие соединения с сервером валидации.

**Ожидаемый результат:**

Система должна показывать корректное recoverable-сообщение о сетевой проблеме, например `Connection lost. Please check your internet connection and try again`, и позволять повторить submit / CAPTCHA без потери введенных данных.

**Влияние:**

Некорректный error mapping вводит пользователя в заблуждение на финальном шаге регистрации. Это minor-дефект, но он может усилить drop-off при нестабильном мобильном соединении.

### BUG-06: [UI/Mobile] Registration form is horizontally cut off on narrow screens

| Поле | Значение |
| --- | --- |
| Severity | High |
| Priority | High |
| Область | Registration / Mobile Layout / Responsive UI |
| Статус | Reproducible |

**Шаги воспроизведения:**

1. Открыть `https://goldbet.io/registration` или актуальное зеркало.
2. Включить эмуляцию мобильного устройства или открыть страницу на экране шириной менее 400px.
3. Проверить форму регистрации и блок 18+ / Terms consent.

**Фактический результат:**

Форма не адаптируется под узкий viewport. Правый край контейнера уходит за пределы видимой области, из-за чего обрезается текст legal consent (`Я подтверждаю, что мне не менее 18...`) и часть элементов управления.

**Ожидаемый результат:**

Форма должна полностью помещаться в мобильный viewport без горизонтального обрезания. Текст legal consent и все элементы управления должны переноситься / масштабироваться в пределах экрана.

**Влияние:**

Для мобильного gambling-трафика сломанная верстка на регистрации напрямую снижает конверсию: пользователь может не видеть обязательный legal text или часть controls на финальном шаге создания аккаунта.

## Короткие ответы

### Что здесь может быть самым дорогим багом для бизнеса?

Самым дорогим багом для бизнеса на этой странице был бы отказ ключевых conversion gates: бесконечный цикл CAPTCHA для реального пользователя или неработающий social login, особенно Google / Telegram.

Для онлайн-казино это напрямую бьет по первым депозитам. Социальный логин дает высокую конверсию на мобильных устройствах: пользователь уже прогрет, пришел из рекламы или партнерского источника и хочет быстро попасть в продукт. Если кнопка Google/Telegram не работает или CAPTCHA не дает пройти дальше, значительная часть горячего трафика не будет вручную вводить E-mail, пароль и проходить дополнительные подтверждения.

Поэтому самые дорогие regression-зоны для этой формы: CAPTCHA availability, social login availability, корректная работа при privacy blockers / Brave / AdBlock и понятный recoverable state при обрыве сети на финальном шаге регистрации.

### Что бы я улучшил в UI/UX, даже если формально багов нет?

- Если позволяет Security Policy, добавить нейтральный статус лоадера / проверки для CAPTCHA-flow, чтобы снизить UX-трение без раскрытия деталей валидации.
- Завести отдельный smoke/regression check для CAPTCHA и social login providers (Google/Telegram), потому что эти точки сильнее всего влияют на мобильную конверсию и первые депозиты.
- Добавить обязательную mobile viewport проверку для ширины менее 400px: форма регистрации, 18+ consent и основные controls не должны обрезаться или уходить за пределы экрана.
- Заменить снимаемый `18+ / Terms consent` checkbox на статичный legal consent text под кнопкой `Sign Up`, чтобы исключить invalid state со снятой предзаполненной галочкой.
- Для вкладки `Один клик` сохранить текущий stateless-подход как ожидаемый behavior, но визуально отделить этот метод от ручного ввода, чтобы пользователь понимал смену парадигмы регистрации.
- Для редиректа между доменами добавить нейтральный брендированный loader без технической терминологии: например, анимацию логотипа, `Securely connecting...` или `Loading gaming lobby...`.
- Добавить более заметное состояние выбранного бонуса и объяснение, можно ли изменить бонус/валюту после регистрации.
- Добавить доступные имена icon-only кнопкам социальных методов авторизации.

## Итог

Форма содержит верхний блок социального входа (`Facebook`, `Google`, `Telegram`) и основные вкладки регистрации: E-mail, Phone и `Один клик`; также поддерживает бонус, валюту, промокод и переход к авторизации. Ключевые подтвержденные дефекты: BUG-03 с User Enumeration через форму регистрации, BUG-04 с client-side silent feedback при снятом 18+ / Terms consent, BUG-05 с некорректной CAPTCHA-ошибкой при потере сети и BUG-06 со сломанной мобильной версткой на узких экранах. Дополнительно зафиксированы BUG-01 по accessibility debt и BUG-02 по loading-state редиректа.
